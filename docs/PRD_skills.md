# PRD Skills System

## Purpose
Provide project-local (not global) agent skills following the Router-Skill pattern, and
guarantee that the two debaters use *different* skills so the debate cannot collapse into
identical behavior (lecture §8.3 #2).

## Background
A skill is a reusable capability described by a `skill.md` file and implemented by a
`skill.py` module exposing a uniform `execute(payload) -> SkillResult` interface. The
`RouterSkill` reads the descriptions and selects a skill by relevance, keeping the agent
`System Prompt` small (context-engineering: load only the relevant skill, not every
description). Skills are owned by the project, not registered as global Claude agent skills.

## Requirements
- Every skill subclasses `BaseSkill` and returns a `SkillResult` (`ok`/`fail`), never raising
  for expected failures, so callers can degrade gracefully.
- Skills are project-local: each lives in its own directory with `skill.md` + `skill.py`.
- `web_search` is wired into the Judge and both debaters because internet evidence is mandatory.
- **Each son owns a distinct distinctive skill** so the two debaters cannot collapse together:
  - Pro → `argument_builder` (constructs the affirmative turn)
  - Con → `rebuttal_builder` (constructs the rebuttal turn)
- The debater routes its fully-rendered system prompt through its distinctive skill via a
  `prompt` passthrough, so the turn keeps exactly **one** LLM call and that call still flows
  through `ApiGatekeeper`. Absent a distinctive skill, the agent falls back to a direct call.
- Judge-side analysis skills (`fact_check`, `stance_check`, `respect_check`, `rebuttal_check`)
  remain available as reusable, independently tested mechanisms.

## Interface
- `BaseSkill.execute(payload: dict) -> SkillResult`
- `RouterSkill.select(query: str) -> BaseSkill`
- `ArgumentBuilderSkill.execute({"prompt": str})` / `RebuttalBuilderSkill.execute({"prompt": str})`
- `run_distinctive_skill(agent, prompt) -> tuple[str | None, dict[str, SkillResult]]`

## Design rationale
- **Distinct skills over a single shared one**: a shared generator would let both sons drift
  to the same wording/stance; separate affirmative vs. rebuttal skills make the opposition
  structural, not just prompt-deep.
- **Prompt passthrough over a second LLM call**: the carefully tuned debater prompt (limits,
  stance rules, anti-echo, history) is preserved and cost stays at one call per turn.
- **Alternative considered**: feeding skill output back into the prompt — rejected because the
  `@final` Template Method fixes `_build_prompt → _execute_skills → _call_llm`, and adding a
  second round-trip doubled cost without improving quality.

## Acceptance
- `ProDebaterAgent.distinctive_skill == "argument_builder"` and
  `ConDebaterAgent.distinctive_skill == "rebuttal_builder"`, and the two differ.
- A wired Pro/Con turn is generated by its distinctive skill and recorded in
  `last_skill_results`.
- The `prompt` passthrough completes the supplied prompt directly; the field-based path is
  unchanged.
- A debater with no distinctive skill wired falls back to a direct LLM call.
- All skills return structured `SkillResult` data and are covered by unit tests.
